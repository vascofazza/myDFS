<h5><a id="user-content-bozza-aggiornata-al-16-gennaio-2015" class="anchor" aria-hidden="true" href="#bozza-aggiornata-al-16-gennaio-2015"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>(bozza aggiornata al 16 gennaio 2015)</h5>
<p><strong>Scopo:</strong></p>
<p>Sviluppo di un semplice file system distribuito. Il file system ha l'equivalente delle primitive <em>open, close, read</em> e <em>write</em>.<br>
I dati dei file sono mantenuti in una copia primaria da parte del server ed in copie secondarie da parte dei client.<br>
La coerenza mantenuta attraverso un meccanismo di invalidazione (attivato dal server) delle cache dei client.<br>
Il server deve supportare sia la modalit√† multi-processo che multi-thread selezionabile da file di configurazione.<br>
Nel seguito xxx.yyy.<a href="http://www.zzz" rel="nofollow">www.zzz</a> sta ad indicare un indirizzo IPv4 nella forma "dotted decimal" (<em>e.g.</em>, 150.146.2.201) ma l'indirizzo<br>
del server deve poter essere specificato anche con il nome mnemonico (<em>e.g.,</em> twin.iac.rm.cnr.it).</p>
<p>Caratteristiche:</p>
<ul>
<li>
<p>Il server gestisce in maniera concorrente le richieste dei client__</p>
</li>
<li>
<p>Il server in ascolto, per <em>default,</em> sulla porta TCP <strong>6020</strong>. La porta pu essere modificata da file di configurazione o da linea comandi tramite l'opzione <strong>-P</strong></p>
</li>
<li>
<p>Il client utilizza una libreria (<em>libmydfs.a)</em> che supporta le seguenti quattro primitive:</p>
<ul>
<li>
<p>MyDFSId *mydfs_open(xxx.yyy.<a href="http://www.zzz" rel="nofollow">www.zzz</a>, char *<strong>nomefile</strong>, int <strong>modo</strong>, int *<strong>err</strong>)</p>
</li>
<li>
<p>int mydfs_close(<strong>MyDFSId</strong>)</p>
</li>
<li>
<p>int mydfs_read(<strong>MyDFSId</strong>, int <strong>pos</strong>, void *<strong>ptr</strong>, unsigned int <strong>size</strong>)</p>
</li>
<li>
<p>int mydfs_write(<strong>MyDFSId</strong>, int <strong>pos</strong>, void *<strong>ptr</strong>, unsigned int <strong>size</strong>)
MyDFSId un identificatore del file. Si pu assumere una definizione del tipo<br>
typedef struct mydfsid { xxx; yyy; zzz; } <strong>MyDFSId</strong>;<br>
dove per il numero di elementi della struttura (ed il loro tipo) non esistono vincoli<br>
(ovviamente la struttura deve avere almeno un elemento).<br>
Per quanto riguarda il nome del file (parametro <strong>nomefile</strong>) deve essere un path specificato rispetto ad una directory di base predefinita nel<br>
file di configurazione (nello stile Unix, con separatore <strong>/</strong>).<br>
Per quanto riguarda il <strong>modo</strong>, devono essere supportate le seguenti modalit (per il significato vedere la documentazione sulla <em>open</em> Unix):</p>
</li>
<li>
<p>O_RDONLY, O_WRONLY, O_RDWR, O_CREAT, O_TRUNC, O_EXCL,  O_EXLOCK
per <em>default</em> i file sono condivisi in lettura.<br>
I parametri ptr e size hanno lo stesso significato della <em>read</em> Unix.<br>
Il parametro pos indica la posizione rispetto cui effettuare l'operazione di lettura o scrittura.<br>
Pu assumere i valori SEEK_SET, SEEK_CUR, o SEEK_END con gli stessi significati che hanno per la _fseek.</p>
</li>
<li>
<p>Il valore di ritorno della mydfs_open :</p>
<ul>
<li>un puntatore ad un MyDFSId se l'operazione andata a buon fine;</li>
<li>NULL in caso di errore. In questo caso la locazione puntata da <strong>err</strong> conterr√†
<ul>
<li>-1 in caso il file aperto in scrittura da un altro client;</li>
<li>-2 in caso il server non sia raggiungibile;</li>
<li>-3 in caso di errore sul file (ad esempio file aperto in sola lettura che non esiste);</li>
</ul>
</li>
</ul>
</li>
<li>
<p>Il valore di ritorno della mydfs_close :</p>
<ul>
<li>0 se l'operazione andata a buon fine;</li>
<li>-1 in caso di errore.</li>
</ul>
</li>
<li>
<p>Il valore di ritorno della mydfs_read :</p>
<ul>
<li>0 se il file stato completamente letto;</li>
<li>-1 in caso di errore;</li>
<li>&gt;0 ed uguale al numero di byte effettivamente letti.</li>
</ul>
</li>
<li>
<p>Il valore di ritorno della mydfs_write :</p>
<ul>
<li>-1 in caso di errore;</li>
<li>&gt;= 0 ed uguale al numero di byte effettivamente scritti.</li>
</ul>
</li>
</ul>
</li>
<li>
<p>La libreria implementa un meccanismo di _caching _sia in lettura, sia in scrittura.</p>
<ul>
<li>
<p>In lettura il client accede i dati dalla propria cache, se disponibili. Altrimenti richiede al server i dati necessari. Ogni richiesta viene effettuata in unit di un blocco logico la cui dimensione 65536 byte. Se il file pi piccolo oppure se l'ultimo frammento del file pi piccolo viene ritornato il numero di byte effettivamente disponbile.</p>
<ul>
<li>Per mantenere la coerenza, **prima **di ogni operazione di lettura dalla cache il client deve controllare se ha ricevuto dal server un messaggio di invalidazione della cache (vedi punto successivo).</li>
<li>Risulta quindi necessario che ogni client abbia una connessione di <em>controllo</em> su cui ricevere i messaggi di invalidazione.</li>
</ul>
</li>
<li>
<p>In scrittura, le operazioni sono effettuate sulla cache locale e sono inviate al server quando viene invocata la <em>mydfs_close.</em></p>
</li>
</ul>
</li>
<li>
<p>Quando un file viene aperto in scrittura, viene assegnato un lock al client che ha invocato la mydfs_open. Se il file gi aperto in scrittura da parte di un altro client viene ritornato un errore specifico (vedi descrizione della mydfs_open). Se un file viene aperto in scrittura tutte le copie tenute in cache dai client vengono invalidate. L'invalidazione avviene tramite un messaggio inviato dal server ai client che hanno una copia del file in cache. A questo scopo il server mantiene una tabella che contiene una riga per ogni file aperto ed in  cui sono elencati tutti i client che hanno il file aperto in lettura.</p>
</li>
<li>
<p>Per evitare che un client che abbia aperto il file in scrittura lo tenga bloccato anche se termina in maniera anomale (senza effettuare la mydfs_close), deve essere implementato un meccanismo di _heart-beating: _periodicamente (ad esempio ogni minuto) il server contatta i client che tengono file aperti in scrittura inviando un messaggio sulla connessione di controllo. Se il client non risponde entro un certo intervallo di tempo, il server assume che il client non sia pi attivo e riprende il lock sul file.</p>
</li>
<li>
<p>I file possono essere creati tramite la mydfs_open oppure essere pre-esistenti.</p>
</li>
<li>
<p>Il server ed il client devono offrire le stesse funzionalit sotto Linux e Windows ed essere interoperabili (server Linux, client Windows oppure server Windows, client Linux oltre a, ovviamente, configurazioni solo Linux o solo Windows)</p>
<ul>
<li>la scelta della modalit (multi-thread o multi-processo) del server deve poter essere effettuata all'atto della<br>
partenze (tramite opzioni e/o lettura di un file di configurazione).</li>
<li>le scelte sul numero di thread o processi e sulla porta su cui rimane in attesa il server devono essere modificabili tramite opzioni e/o lettura di un file di configurazione.</li>
</ul>
</li>
<li>
<p>Il file system non fa distinzione tra file binari o di testo.</p>
</li>
<li>
<p>Il server deve funzionare in modalit daemon sotto Unix/Linux.</p>
</li>
</ul>
<p>Suggerimenti:</p>
<ul>
<li>Definire un semplice protocollo per lo scambio delle informazioni tra client e server (comando-risposta);</li>
<li>Utilizzare una connessione distinta per i dati e una per il controllo;</li>
<li>Per il meccanismo di caching utilizzare i concetti di file mapping ( un suggerimento non vincolante);</li>
<li>Per il meccanismo di _heart-beating _usare un thread separato oppure un meccanismo tipo <em>signal;</em></li>
<li>client e server <strong>devono</strong> essere provati NON solo in locale (sullo stesso sistema) ma anche su sistemi distinti.</li>
</ul>
<p>Il progetto pu√≤ essere realizzato in gruppi di <strong>due</strong> persone.</p>
